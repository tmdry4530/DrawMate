## 2. 한 줄 요약

**DrawMate는 웹툰·일러스트·애니메이션 분야의 창작자와 어시스턴트를 연결하기 위해, 태그 기반 탐색·커스터마이징 포트폴리오·실시간 메시징을 제공하는 전문 구인구직 플랫폼이다.**

---

## 3. 서비스 비전 & 미션

### 3.1 비전 (Vision)
국내 창작 보조 인력 시장에서 **“분야 특화형(Vertical) 포트폴리오 기반 매칭 표준”** 이 되는 것.

### 3.2 미션 (Mission)
기존 커뮤니티형 구인구직의 비정형성과 탐색 비효율을 줄이고,  
창작자와 어시스턴트가 **더 빠르게 발견되고, 더 정확하게 비교되며, 더 자연스럽게 연결되는 경험**을 제공한다.

### 3.3 핵심 가치 제안 (Value Proposition)
- **정교한 탐색**: 분야(Field), 스킬(Skill), 툴(Tool), 스타일(Style) 중심의 구조화된 탐색
- **표현력 있는 포트폴리오**: 템플릿 기반 커스터마이징으로 작업 성향과 강점 표현
- **연결 속도 개선**: 실시간 메시징과 알림으로 첫 접점까지의 시간을 단축
- **비관여 방문자 유입**: 포트폴리오 감상 목적의 일반 방문자도 유입 가능한 공개형 구조

## 4. 문제 정의

현재 그림/웹툰 어시스턴트 구인구직은 주로 네이버 카페 같은 커뮤니티 게시글 중심으로 이뤄진다. 이 구조는 “게시글 작성과 검색”에는 적합하지만, **전문 인력 탐색과 비교**에는 비효율적이다.

### 4.1 분야 미분류 문제
- 웹툰, 일러스트, 애니메이션, 배경, 캐릭터, 채색, 후보정 등 세부 전문 분야가 충분히 구조화되지 않는다.
- 게시판 또는 제목/본문 키워드에 의존해 검색해야 하므로, 적합한 인재 또는 포지션을 찾는 비용이 높다.

### 4.2 포트폴리오 커스터마이징 부재
- 일반 게시글 첨부 방식은 작업 성격과 강점을 체계적으로 보여주기 어렵다.
- 작품 배열, 커버 강조, 소개 구조, 태그 정리 같은 포트폴리오적 표현이 제한적이다.

### 4.3 매번 글 재작성 필요
- 구직자는 지원 포지션이나 시즌마다 게시글을 새로 작성·복사·수정해야 한다.
- 이전 작업물, 소개, 가용 시간, 연락처, SNS 링크, 작업 조건 등을 반복 입력해야 한다.

### 4.4 구인자의 비효율적 탐색
- 구인자는 많은 게시글을 일일이 열어보며 조건을 수동 비교해야 한다.
- 정렬/필터/북마크/비교 체계가 부족하면 탐색 시간이 길어지고, 적합 후보를 놓칠 가능성이 커진다.

### 4.5 신뢰 가능한 전문 매칭 부재
- “누가 어떤 작업을 잘하는지”가 텍스트 중심 글만으로는 충분히 드러나지 않는다.
- 실제 작업물, 태그, 작업 가능 시간, 프로필 정보, 커뮤니케이션 반응성이 한 화면 흐름에서 연결되지 않는다.

### 4.6 DrawMate가 해결하는 방식
DrawMate는 커뮤니티 게시글 중심 흐름을 **“구조화된 포트폴리오 + 탐색 + 연결”** 구조로 전환한다.  
즉, 글을 찾는 방식이 아니라 **사람과 작업 역량을 탐색하는 방식**으로 경험을 재설계한다.

## 5. 타겟 유저 페르소나

## 5.1 페르소나 A — 구직자(어시스턴트)
- **유형**: 웹툰/일러스트/애니메이션 어시 구직 중인 사람
- **배경**: 프리랜서, 취업 준비생, 프로젝트 단위 협업 희망자, 현업 부업 인력
- **목표**
  - 자신의 작업물과 강점을 잘 보여주고 싶다.
  - 자신에게 맞는 구인자를 빠르게 만나고 싶다.
  - 매번 새 글을 쓰지 않고 포트폴리오를 재사용하고 싶다.
- **핵심 불편**
  - 게시글만으로는 전문성을 설득하기 어렵다.
  - 어떤 태그/키워드로 구인자를 찾을지 불명확하다.
  - 연락이 와도 내 작업 성향과 맞는 제안인지 판별이 어렵다.
- **성공 조건**
  - 포트폴리오 등록 후 검색/노출이 잘 된다.
  - 프로필과 작업 가능 시간 기반으로 적합한 제안을 받는다.
  - 메시지 응답과 협업 연결이 빠르다.

## 5.2 페르소나 B — 구인자(작가/PD)
- **유형**: 어시스턴트를 찾는 작가, 프로덕션, 스튜디오 PD
- **배경**: 연재 일정이 있는 웹툰 작가, 프로젝트형 일러스트 제작자, 애니메이션/콘텐츠 제작 팀
- **목표**
  - 필요한 작업군에 맞는 사람을 빠르게 찾고 싶다.
  - 작업물과 태그를 보고 후보를 효율적으로 비교하고 싶다.
  - 연락 전 기본 정보와 작업 가능 조건을 확인하고 싶다.
- **핵심 불편**
  - 게시글이 많아도 필요한 분야별 후보를 찾기 어렵다.
  - 포트폴리오 형식이 제각각이라 비교가 어렵다.
  - 연락 후에야 가용 시간, 작업 툴, 스타일 적합성이 드러난다.
- **성공 조건**
  - 필터와 검색만으로 shortlist를 빠르게 만든다.
  - 북마크/정렬/상세 비교로 탐색 시간이 단축된다.
  - 메시지 전송 후 평균 응답 시간이 짧다.

## 5.3 페르소나 C — 비관여 방문자
- **유형**: 포트폴리오 구경 목적의 일반 사용자
- **배경**: 그림 커뮤니티 이용자, 예비 창작자, 팬, 잠재적 미래 구인자
- **목표**
  - 다양한 포트폴리오를 편하게 둘러보고 싶다.
  - 특정 스타일이나 분야의 작업물을 찾아보고 싶다.
- **핵심 불편**
  - 기존 커뮤니티는 작품 감상형 탐색 경험이 약하다.
  - 포트폴리오가 아닌 게시글 형태라 탐색 동기가 약해진다.
- **성공 조건**
  - 홈/탐색에서 매력적인 포트폴리오를 쉽게 발견한다.
  - 태그 중심으로 관심 분야를 따라가며 둘러볼 수 있다.
  - 향후 구직자 또는 구인자로 전환될 수 있다.

### 6.1 경쟁 해석 요약
- **네이버 카페**는 커뮤니티 도달성은 높지만 전문 인력 탐색 UX가 약하다.
- **크몽**은 거래 안정성은 강하지만 DrawMate가 노리는 “창작 보조 인력” 맥락에 대한 세분화가 제한적이다.
- **ArtStation**은 포트폴리오 완성도는 매우 강하지만, 현지화된 어시 채용 경험으로 바로 이어지진 않는다.
- **X**는 확산과 발견에는 유리하지만 검색 결과의 구조화와 비교 가능성이 낮다.

## 7. 제품 목표 (Product Goals)

### 7.1 비즈니스 목표
- 창작 보조 인력 채용/구직의 대표 온라인 진입점 확보
- 포트폴리오 기반 반복 방문 유도
- 구인자와 구직자 양측의 네트워크 효과 형성

### 7.2 사용자 목표
- 구직자는 “한 번 잘 만든 포트폴리오”로 반복 기회를 얻는다.
- 구인자는 “짧은 시간 안에 적합 후보를 좁혀” 연락한다.
- 일반 방문자는 “작품 감상 → 팔로우/북마크/가입”으로 자연스럽게 전환된다.

### 7.3 MVP 성공 정의
- 사용자가 회원가입 후 **프로필 작성 → 포트폴리오 등록 → 탐색/연락** 흐름을 막힘 없이 수행한다.
- 구인자는 태그/검색/정렬만으로 shortlist를 만들 수 있다.
- 첫 메시지 이후 평균 응답 시간과 실제 협업 연결률이 기존 커뮤니티 대비 유의미하게 개선된다.

## 8. MVP 범위

아래 기능은 가이드 문서 기준 핵심 기능을 중심으로 정리했으며, 서비스 운영상 필수인 기반 기능을 함께 포함한다.

### 8.1 회원가입 / 로그인 (Auth)
- 이메일 및 소셜 로그인 기반 가입/인증
- 최소 정보로 빠르게 온보딩 시작
- 역할(구직자/구인자) 선택을 통해 이후 흐름 분기
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.2 포트폴리오 CRUD
- 포트폴리오 생성, 조회, 수정, 삭제
- 초안(Draft)과 공개(Published) 상태 관리
- 제목, 소개, 작업 가격/기간 등 핵심 메타데이터 입력
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.3 이미지 업로드
- 최대 20장 업로드
- 드래그 앤 드롭 업로드
- 이미지 순서 변경, 캡션 입력, 커버 이미지 지정
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.4 태그 시스템
- 분야(Field), 스킬(Skill), 툴(Tool), 스타일(Style) 카테고리 제공
- 다중 태그 선택 및 편집
- 포트폴리오와 프로필에 태그 연결
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.5 검색 / 필터 / 정렬
- 키워드 검색
- 태그 기반 다중 필터
- 최신순 / 인기순 / 가격순 정렬
- 구인자가 탐색 시간을 줄일 수 있는 핵심 기능
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.6 포트폴리오 템플릿
- 3종 이상의 템플릿 제공
- 커버 이미지/색상 수준의 기본 커스터마이징 지원
- “게시글”이 아니라 “포트폴리오”로 보이도록 표현력 강화
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.7 사용자 프로필
- 기본 정보, 소개글, SNS 링크, 작업 가능 시간
- 역할 기반 정보 차등 노출
- 포트폴리오와 함께 신뢰 보조 정보 제공
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.8 1:1 메시징 / 채팅
- 텍스트 메시지
- 이미지 첨부
- 읽음 확인
- 포트폴리오 상세에서 자연스럽게 연락으로 이어지는 구조
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.9 알림 시스템
- 새 메시지
- 포트폴리오 관련 활동
- 주요 상태 변화에 대한 Push + In-app 알림
- 정확한 이벤트 범위는 기능 명세서에서 확정
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

### 8.10 북마크
- 관심 포트폴리오 저장
- 구인자의 shortlist 작성 지원
- 재방문 시 빠른 비교/검토 가능
- **상세 UI는 `DrawMate/Wireframe/` 와이어프레임 참조**

## 10. KPI (초기 가설 목표치)

> 아래 목표치는 베타~초기 출시 구간을 가정한 **초기 가설 수치**이며, 실제 목표는 채널 전략/베타 성과에 따라 조정한다.

### 10.1 MAU (Monthly Active Users)
- 정의: 월 1회 이상 로그인 또는 공개 포트폴리오 탐색 활동을 수행한 사용자 수
- 초기 목표: 출시 3개월 내 3,000 MAU

### 10.2 포트폴리오 등록수
- 정의: 공개 상태(Published) 포트폴리오 누적 수
- 초기 목표: 출시 3개월 내 1,000건

### 10.3 매칭 성공률
- 정의: 첫 메시지 발생 후 실제 협업 시작으로 이어진 비율
- 초기 목표: 15% 이상
- 비고: 정확한 “성공” 이벤트 정의는 추후 운영 기준 합의 필요

### 10.4 평균 응답 시간
- 정의: 첫 문의 메시지 발송 후 첫 응답까지의 중앙값(Median)
- 초기 목표: 24시간 이내

### 10.5 리텐션 (Retention)
- 정의: 가입 후 일정 기간 내 재방문/재활동 비율
- 초기 목표:
  - D7 Retention 35% 이상
  - D30 Retention 20% 이상

## 11. 마일스톤 (MVP 10주 기준)

| 기간 | 목표 | 주요 산출물 |
|---|---|---|
| 1주차 | PRD 확정 및 범위 잠금 | PRD, 핵심 사용자 시나리오 |
| 2주차 | 유저 플로우/IA 정리 | 플로우, IA, 화면 목록 |
| 3주차 | 기능 명세서 상세화 | Feature Spec |
| 4주차 | ERD / API 초안 확정 | ERD, API Spec |
| 5~6주차 | 프론트엔드 핵심 화면 개발 | 홈/탐색/상세/에디터/프로필 |
| 7~8주차 | 백엔드 연동 및 실시간 기능 구현 | Auth, DB, Storage, Messaging, Notification |
| 9주차 | QA / 버그 수정 / 베타 준비 | 테스트 케이스, 운영 점검 |
| 10주차 | 폐쇄 베타 및 개선 반영 | 베타 피드백, 런칭 후보 빌드 |

### 11.1 8~12주 범위 운영 원칙
- **8주 압축형**: 디자인/개발 병렬, 기능 우선
- **10주 표준형**: 명세 → 구현 → QA 균형
- **12주 안정형**: QA/분석/운영 준비 강화

## 12. 비기능 요구사항 (PRD 레벨)

### 12.1 사용성
- 포트폴리오 등록은 비전문 사용자도 10분 내 초안 작성 가능해야 한다.
- 구인자는 필터/검색으로 3분 내 shortlist 작성 가능해야 한다.

### 12.2 성능
- 홈/탐색 첫 체감 로딩은 빠르게 유지되어야 한다.
- 이미지 중심 서비스 특성상 썸네일/원본 분리와 최적화가 필요하다.

### 12.3 신뢰와 안전
- 프로필/메시지/알림 흐름은 스팸 방지와 신고 정책을 고려해야 한다.
- 공개 범위와 개인정보 노출 정책을 조기에 정의해야 한다.

### 12.4 확장성
- 향후 리뷰/평판, AI 매칭, 결제 연동이 붙더라도 데이터 구조와 사용자 흐름이 크게 흔들리지 않아야 한다.

## 13. 주요 가정과 오픈 이슈

1. MVP에서는 “빠른 발견과 연결”이 최우선이며, 결제/평판은 후속으로 둔다.  
2. 포트폴리오 템플릿은 3종 이상 제공하되 과도한 자유도보다 작성 완료율을 우선한다.  
3. 알림 이벤트의 정확한 범위는 기능 명세서에서 확정한다.  
4. “매칭 성공률” 측정 이벤트 정의(메시지 → 협업 확정)는 운영 정책과 함께 설계해야 한다.  
5. 실제 화면 구조와 컴포넌트 배치는 문서상 추정하지 않고, 반드시 `DrawMate/Wireframe/` 실물을 기준으로 맞춘다.
