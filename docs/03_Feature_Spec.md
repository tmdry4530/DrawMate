# DrawMate — 기능 명세서 (Feature Specification)

버전: Draft v0.1  
작성일: 2026-03-21  
기반 문서: `DrawMate_Project_Guide.docx` 3.3 기능 명세서, PRD v0.1, 유저 플로우 & IA v0.1

---

## 0. 중요 전제

- 본 문서는 **개발 가능한 수준의 요구사항** 을 정의한다.
- 실제 버튼 위치, 탭 순서, 모달/드로어 사용 여부, 카드/리스트 레이아웃은 `DrawMate/Wireframe/` 실물 확인이 필요하다.
- 따라서 모든 기능 섹션의 **와이어프레임 참조 문구는 필수**이며, 구체적 UI 요소 배치는 본 문서에서 임의 확정하지 않는다.

---

## 1. 공통 명세 원칙

### 1.1 우선순위 정의
- **MVP**: 베타/초기 런칭에 반드시 포함
- **v1.1**: MVP 이후 빠른 개선 릴리즈
- **v2.0**: 추천/결제/고도화 중심 확장 기능

### 1.2 공통 비기능 요구사항
- 프론트엔드와 API의 모든 데이터 계약은 **TypeScript 타입** 기준으로 관리한다.
- 서버 권한 검사는 UI 제약과 별개로 **항상 서버에서 재검증**한다.
- 모든 사용자 입력은 서버에서 **길이, 형식, 소유권, 허용 값(enum)** 검사를 수행한다.
- 오류 메시지는 사용자용 메시지와 운영용 로그를 분리한다.
- 파일 업로드/메시지 전송/북마크 토글 등은 **멱등성(idempotency)** 을 고려한다.

### 1.3 공통 상태 용어
- `Draft`: 작성 중, 공개 미노출
- `Published`: 공개 노출 상태
- `Archived`: 공개 목록 미노출, 내부 보관
- `Unread`: 읽지 않은 메시지/알림
- `Muted`: 알림만 제한하고 데이터 접근은 유지

## 2.1 포트폴리오 CRUD

### 기능명
포트폴리오 CRUD (Create / Read / Update / Delete)

### 우선순위
MVP

### 사용자 스토리
- **구직자로서**, 내 작업물과 소개를 담은 포트폴리오를 만들고 싶다, **왜냐하면** 구인자가 내 역량을 글이 아닌 결과물 중심으로 판단할 수 있어야 하기 때문이다.
- **구직자로서**, 기존 포트폴리오를 수정·재사용하고 싶다, **왜냐하면** 매번 새 게시글을 다시 작성하는 비용을 줄이고 싶기 때문이다.

### 상세 요구사항

#### 입력
- `title`: 2~80자, 필수
- `summary`: 10~300자, 필수
- `description`: 20~5,000자, 선택이지만 발행 시 권장
- `templateId`: 시스템 제공 템플릿 중 1개, 필수
- `startingPriceKrw`: 0 이상 정수, 선택
- `durationDays`: 1~365, 선택
- `visibility`: `public` 또는 `unlisted`, 기본 `public`
- `status`: 생성 시 무조건 `draft`, 발행 시 `published`

#### 출력
- 포트폴리오 식별자
- 생성/수정/발행 시각
- 현재 상태(`draft/published/archived`)
- 공개 URL(slug 기반) 또는 상세 화면 진입 키

#### 유효성 검증
- 제목/요약은 공백만 입력 불가
- 발행 시 최소 1장의 이미지 필요
- 발행 시 최소 1개의 `field` 카테고리 태그 필요
- 사용자 본인 소유 포트폴리오만 수정/삭제 가능
- 삭제는 소프트 삭제 또는 `archived` 전환 정책 중 하나로 서버에서 일관 처리

#### 제약조건
- MVP 기준 사용자당 **최대 10개 포트폴리오**
- 동일 사용자의 `draft` 와 `published` 는 자유롭게 공존 가능
- `published` 상태만 HOME/EXPLORE/검색 결과에 노출
- `archived/deleted` 상태는 공개 검색 결과에서 제외

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 포트폴리오 에디터, 포트폴리오 상세, 프로필 관련 화면 참조
- 구체적 UI 요소 배치 및 CTA 위치는 와이어프레임 확인 필요

### 엣지케이스
- 필수 이미지 없이 발행 시도
- 비공개 초안 URL 직접 접근
- 이미 북마크된 포트폴리오 삭제
- 수정 중 세션 만료
- 두 탭에서 같은 포트폴리오 동시 수정

### 수용 기준 (Acceptance Criteria)
1. 유효한 최소 입력으로 포트폴리오 Draft가 생성된다.
2. 작성자는 본인 포트폴리오만 수정/삭제할 수 있다.
3. 필수 이미지와 필수 태그가 없으면 `published` 전환이 거부된다.
4. `published` 상태의 포트폴리오만 공개 탐색 결과에 노출된다.
5. 삭제 또는 보관된 포트폴리오는 공개 상세 접근 시 적절한 안내가 표시된다.

## 2.2 이미지 업로드

### 기능명
포트폴리오 이미지 업로드 / 정렬 / 커버 지정

### 우선순위
MVP

### 사용자 스토리
- **구직자로서**, 여러 장의 작업 이미지를 순서대로 업로드하고 싶다, **왜냐하면** 내가 보여주고 싶은 작업 흐름과 대표작을 직접 통제해야 하기 때문이다.

### 상세 요구사항

#### 입력
- 파일 형식: `image/jpeg`, `image/png`, `image/webp`
- 파일 개수: 포트폴리오당 최대 20장
- 파일 용량: 1장당 최대 10MB
- `caption`: 이미지당 0~200자, 선택
- `sortOrder`: 0 이상 정수
- `isCover`: 포트폴리오당 최대 1장만 `true`

#### 출력
- 업로드 성공 이미지 목록
- 각 이미지의 `imageId`, 저장 경로, 가로/세로 크기, 썸네일/웹용 URL
- 실패 파일별 오류 메시지

#### 유효성 검증
- MIME 타입 위조 방지를 위해 서버에서 실제 파일 헤더 검사
- 이미지 수 제한(20장) 초과 시 전체 또는 초과분 거부 정책을 명시적으로 처리
- 동일 `sortOrder` 중복 불가
- 커버 이미지는 정확히 0 또는 1개
- 업로드 직후 미리보기 가능한 상태여야 함

#### 제약조건
- 드래그 앤 드롭 업로드 지원
- 다중 업로드 중 일부 실패 시 성공 항목은 유지하고 실패 항목만 재시도 가능해야 함
- 서버는 업로드 후 웹 표시용 최적화본을 생성해야 함
- 포트폴리오 삭제 시 연결된 이미지 객체도 정리 정책 필요

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 포트폴리오 에디터, 이미지 배치/정렬 화면 참조
- 썸네일 리스트/그리드/드롭존 구조는 와이어프레임 확인 필요

### 엣지케이스
- 네트워크 중단으로 부분 업로드만 완료
- 동일 파일 재업로드
- 업로드 후 순서 변경 중 브라우저 새로고침
- 커버 이미지 삭제 시 대체 커버 자동 지정 필요 여부
- 이미지 손상 파일 업로드

### 수용 기준 (Acceptance Criteria)
1. 사용자는 한 포트폴리오에 최대 20장까지 업로드할 수 있다.
2. 사용자는 업로드 후 이미지 순서를 변경할 수 있다.
3. 사용자는 대표 이미지 1장을 커버로 지정할 수 있다.
4. 허용되지 않은 형식 또는 10MB 초과 파일은 업로드가 거부된다.
5. 업로드된 이미지는 에디터 미리보기와 상세 페이지에서 같은 순서로 노출된다.

## 2.3 태그 시스템

### 기능명
태그 시스템 (분야 / 스킬 / 툴 / 스타일)

### 우선순위
MVP

### 사용자 스토리
- **구직자로서**, 내 작업 분야와 툴, 스타일을 태그로 표현하고 싶다, **왜냐하면** 구인자가 텍스트 설명을 모두 읽지 않아도 내 역량을 빠르게 이해할 수 있어야 하기 때문이다.
- **구인자로서**, 태그 체계를 기반으로 후보를 좁히고 싶다, **왜냐하면** 게시글 제목 검색보다 더 정확한 탐색이 필요하기 때문이다.

### 상세 요구사항

#### 입력
- `field`: 예) 웹툰, 일러스트, 애니메이션, 배경, 캐릭터
- `skill`: 예) 선화, 채색, 후보정, 콘티 보조
- `tool`: 예) Photoshop, Clip Studio Paint, Procreate
- `style`: 예) 셀식, 반실사, 캐주얼, SD

#### 출력
- 카테고리별 태그 ID / slug / label
- 선택된 태그 목록
- 검색/필터에 바로 사용 가능한 표준화된 값

#### 유효성 검증
- 사용자 임의 생성 태그는 MVP에서 비허용
- 시스템에서 미리 정의한 canonical tag만 선택 가능
- 태그 slug는 중복 불가
- 카테고리 이외 값 입력 불가

#### 제약조건
- 포트폴리오당 추천 제한:
  - `field`: 1~3개
  - `skill`: 최대 10개
  - `tool`: 최대 5개
  - `style`: 최대 5개
- 비활성 태그는 신규 선택 불가하되, 기존 포트폴리오의 과거 데이터는 유지 가능
- 검색과 표시를 위해 태그는 label + slug + category를 모두 가져야 함

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 태그 선택 컴포넌트, 탐색 필터 바 관련 화면 참조
- 칩(Chip), 멀티셀렉트, 오토컴플리트 여부는 와이어프레임 확인 필요

### 엣지케이스
- 운영자가 태그를 비활성화했지만 기존 포트폴리오에는 남아있는 경우
- 동일 의미의 중복 태그가 taxonomy에 존재하는 경우
- 필수 `field` 태그를 제거한 뒤 발행 저장 시도
- 태그 수 제한 초과

### 수용 기준 (Acceptance Criteria)
1. 사용자는 시스템 정의 태그만 선택할 수 있다.
2. 태그는 category 기준으로 분류되어 저장된다.
3. 필수 `field` 태그가 없으면 발행 검증을 통과하지 못한다.
4. 비활성화된 태그는 새 포트폴리오에 추가할 수 없다.
5. 선택된 태그는 상세 페이지와 탐색 필터 결과에 일관되게 반영된다.

## 2.4 태그 기반 필터링 + 검색 + 정렬

### 기능명
탐색 필터링 / 키워드 검색 / 정렬

### 우선순위
MVP

### 사용자 스토리
- **구인자로서**, 분야·스킬·툴·스타일 태그로 후보를 걸러내고 싶다, **왜냐하면** 필요한 작업군에 맞는 사람만 빠르게 보고 싶기 때문이다.
- **비관여 방문자로서**, 관심 키워드와 스타일로 포트폴리오를 둘러보고 싶다, **왜냐하면** 감상 경험 자체도 플랫폼 가치이기 때문이다.

### 상세 요구사항

#### 입력
- `q`: 검색어, 1~50자
- `fieldTags[]`, `skillTags[]`, `toolTags[]`, `styleTags[]`
- `sort`: `latest`, `popular`, `price_asc`, `price_desc`
- `cursor`: 페이지네이션 커서
- `limit`: 기본 24, 최대 48

#### 출력
- 포트폴리오 카드 목록
- 전체 개수 또는 다음 페이지 존재 여부
- 적용 중인 필터 상태
- 빈 결과 시 추천 행동(필터 초기화, 유사 태그 제안)

#### 유효성 검증
- 존재하지 않는 태그 slug는 무시하지 말고 400 처리 또는 명시적 validation error 반환
- 지원하지 않는 sort 값 거부
- cursor는 서버 서명 또는 안전한 인코딩 값이어야 함

#### 제약조건
- 검색 대상은 `published + public` 포트폴리오만
- 기본 정렬은 `latest`
- 키워드 검색은 제목/요약/설명 중심
- 태그 필터는 **AND within category / OR within same filter bucket** 정책 여부를 명확히 해야 하며, MVP에서는 다음 규칙을 권장:
  - 카테고리 간: AND
  - 같은 카테고리 다중 선택: OR

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 홈 피드, 탐색 페이지, 필터 바 화면 참조
- 필터 바 위치/모바일 접힘 방식/정렬 UI는 와이어프레임 확인 필요

### 엣지케이스
- 결과 0건
- 너무 넓은 검색어로 대량 결과
- 같은 태그를 여러 category로 잘못 전달한 경우
- 공개 직후 아직 검색 인덱스 반영 전 상태
- archived 포트폴리오가 캐시에 남아 보이는 경우

### 수용 기준 (Acceptance Criteria)
1. 사용자는 키워드와 태그를 조합해 검색할 수 있다.
2. 결과 목록은 최신순/인기순/가격순으로 정렬된다.
3. `published + public` 이 아닌 포트폴리오는 결과에 포함되지 않는다.
4. 빈 결과일 때 사용자에게 필터 해제 또는 검색어 수정 안내가 제공된다.
5. 동일 조건 재조회 시 페이지네이션 결과 순서가 안정적이다.

## 2.5 포트폴리오 템플릿

### 기능명
포트폴리오 템플릿 시스템

### 우선순위
MVP

### 사용자 스토리
- **구직자로서**, 내 작업 스타일에 맞는 템플릿을 골라 포트폴리오를 꾸미고 싶다, **왜냐하면** 단순한 게시글보다 훨씬 잘 정리된 인상을 줄 수 있기 때문이다.

### 상세 요구사항

#### 입력
- `templateId`
- `themePreset` 또는 강조 색상
- `coverMode`
- 섹션 노출 옵션(지원 범위 내)

#### 출력
- 선택된 템플릿 기반 미리보기
- 발행 시 템플릿 적용된 공개 상세 화면
- 템플릿 변경 이력 또는 마지막 저장 상태

#### 유효성 검증
- 존재하지 않는 템플릿 ID 불가
- 비활성 템플릿은 신규 적용 불가
- 템플릿 변경 시 기존 콘텐츠 손실 여부를 서버/클라이언트에서 사전 경고

#### 제약조건
- MVP 기준 시스템 제공 템플릿 최소 3종
- 템플릿은 레이아웃 자체보다 **콘텐츠 구조 재배치 + 표현 강조 수준** 에 집중
- 템플릿별 허용 커스터마이징 범위는 동일하지 않을 수 있음
- 상세 UI 구조는 `DrawMate/Wireframe/` 실물 확인 후 확정

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 포트폴리오 에디터, 상세 페이지, 템플릿 선택 흐름 참조
- 각 템플릿의 실제 섹션 구성/비주얼 위계는 와이어프레임 확인 필요

### 엣지케이스
- 템플릿 변경 후 일부 긴 텍스트가 레이아웃을 깨는 경우
- 템플릿 폐기 또는 비활성화
- 발행 후 템플릿 교체로 URL은 유지하되 화면 구조만 변경되는 경우
- 이미지 수가 적거나 지나치게 많아 템플릿 표현이 불리한 경우

### 수용 기준 (Acceptance Criteria)
1. 사용자는 최소 3종 이상의 템플릿 중 1개를 선택할 수 있다.
2. 사용자는 템플릿 변경 후 미리보기를 확인할 수 있다.
3. 템플릿 변경이 기존 콘텐츠를 임의 삭제하지 않는다.
4. 발행된 포트폴리오는 선택된 템플릿 스타일로 렌더링된다.
5. 비활성 템플릿은 신규 포트폴리오에서 선택할 수 없다.

## 2.6 사용자 프로필

### 기능명
사용자 프로필 (기본정보 / 소개 / SNS / 가용시간)

### 우선순위
MVP

### 사용자 스토리
- **구직자로서**, 포트폴리오 외에도 나에 대한 소개와 작업 가능 시간을 보여주고 싶다, **왜냐하면** 구인자가 연락 전에 협업 가능성을 판단할 수 있어야 하기 때문이다.
- **구인자로서**, 내 계정도 기본 신뢰 정보를 갖고 싶다, **왜냐하면** 상대가 어떤 작업자/스튜디오인지 이해할 수 있어야 응답률이 높아지기 때문이다.

### 상세 요구사항

#### 입력
- `displayName`: 2~40자, 필수
- `headline`: 0~80자, 선택
- `bio`: 0~500자
- `avatar`: 이미지 1장, 선택
- `snsLinks[]`: 최대 5개 URL
- `availabilityStatus`: `open`, `busy`, `unavailable`
- `availableHours`: 요일별/시간대별 JSON 구조
- 역할 기반 추가 정보:
  - 구직자: 작업 가능 분야/시간 강조
  - 구인자: 스튜디오/프로덕션 소개(선택)

#### 출력
- 공개 프로필 페이지
- 포트폴리오 상세에 삽입될 요약 프로필
- 프로필 completeness 상태

#### 유효성 검증
- URL 형식 검사
- SNS 링크 도메인 화이트리스트는 MVP에서 비강제, 단 정상 URL만 허용
- displayName은 공백-only 불가
- avatar는 허용 형식/용량 제한 적용

#### 제약조건
- 이메일 주소는 공개 프로필에 직접 노출하지 않음
- 연락은 메시징 시스템을 우선 경유
- availability 정보는 구조화된 값으로 저장해 검색/표시에 재사용 가능해야 함

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 프로필 페이지, 설정, 포트폴리오 상세 내 작성자 영역 참조
- 프로필 편집 폼의 구체 배치는 와이어프레임 확인 필요

### 엣지케이스
- SNS 링크가 죽은 링크인 경우
- 사용자가 아바타 없이도 가입/활동하는 경우
- 공개 범위 축소 후 기존 링크 접근
- 역할 변경 필요성이 생기는 경우(구직자 ↔ 구인자)

### 수용 기준 (Acceptance Criteria)
1. 사용자는 공개 프로필을 생성/수정할 수 있다.
2. 포트폴리오 상세에는 작성자의 핵심 프로필 정보가 표시된다.
3. 이메일 주소는 공개 화면에 노출되지 않는다.
4. 잘못된 SNS URL은 저장되지 않는다.
5. availability 정보는 프로필과 탐색 맥락에서 일관된 형식으로 사용된다.

## 2.7 1:1 메시징 / 채팅

### 기능명
1:1 메시징 / 채팅 (텍스트 + 이미지 첨부)

### 우선순위
MVP

### 사용자 스토리
- **구인자로서**, 적합한 포트폴리오를 본 뒤 바로 연락하고 싶다, **왜냐하면** 다른 채널로 이동하면 응답 시간이 늦어지기 때문이다.
- **구직자로서**, 받은 메시지를 플랫폼 안에서 바로 확인하고 답장하고 싶다, **왜냐하면** 새로운 기회를 놓치지 않아야 하기 때문이다.

### 상세 요구사항

#### 입력
- `conversationId` 또는 상대 `userId`
- `body`: 1~2,000자, 선택 가능
- `attachments[]`: 최대 5개 이미지, 각 10MB 이하
- 메시지 타입: `text`, `image`, `mixed`

#### 출력
- 대화방 생성 또는 기존 대화방 재사용
- 메시지 ID, 작성 시각, 전송 상태, 읽음 상태(대화 단위)
- 상대방 unread 증가 및 알림 이벤트 생성

#### 유효성 검증
- 본인에게 메시지 전송 불가
- 대화 참여자만 메시지 조회 가능
- 빈 문자열 + 첨부 없음 조합은 전송 불가
- 허용되지 않은 첨부 형식 거부
- 상대 사용자 차단 기능은 MVP 범위 밖이지만, 최소한 삭제/비활성 계정 상태는 검사 필요

#### 제약조건
- MVP는 1:1 direct conversation만 지원
- 동일 사용자 쌍에 대해 direct conversation은 1개만 유지
- 읽음 상태는 대화방 단위의 lastRead 기준으로 관리
- 전송 성공 후 optimistic UI와 서버 확정 상태를 동기화

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 메시지 화면, 포트폴리오 상세의 연락 CTA, 대화방 화면 참조
- 입력창/첨부 버튼/읽음 표시의 배치는 와이어프레임 확인 필요

### 엣지케이스
- 이미 대화 중인 상대에게 다시 연락 시 신규 방 생성
- 메시지 전송 중 네트워크 단절
- 이미지 첨부 업로드 성공 후 메시지 insert 실패
- 메시지 목록은 있으나 상대 포트폴리오가 삭제된 경우
- 오프라인 후 재접속 시 unread 불일치

### 수용 기준 (Acceptance Criteria)
1. 로그인 사용자는 포트폴리오 상세 또는 프로필에서 1:1 대화를 시작할 수 있다.
2. 동일 사용자 쌍은 기존 대화방을 재사용한다.
3. 텍스트와 이미지 첨부가 모두 지원된다.
4. 참여자가 아닌 사용자는 해당 대화나 메시지를 볼 수 없다.
5. 읽지 않은 메시지는 메시지 목록과 알림센터에 반영된다.

## 2.8 알림 시스템

### 기능명
알림 시스템 (Push + In-app)

### 우선순위
MVP

### 사용자 스토리
- **구직자로서**, 새 메시지를 즉시 알고 싶다, **왜냐하면** 응답 속도가 기회 손실과 직접 연결되기 때문이다.
- **구인자로서**, 답장이 왔는지 바로 알고 싶다, **왜냐하면** 후보 진행 상황을 놓치지 않아야 하기 때문이다.

### 상세 요구사항

#### 입력
- 시스템 이벤트:
  - `message_received`
  - `message_replied`
  - `bookmark_added`
  - `system_notice`
- 사용자 알림 설정:
  - In-app on/off (필수 on, 개별 유형 mute 가능)
  - Push opt-in/out
  - quiet hours(후속 확장 가능)

#### 출력
- In-app 알림 리스트
- unread 알림 수
- Push 전송 요청(권한 허용 사용자 대상)

#### 유효성 검증
- 동일 이벤트 중복 발송 방지
- 읽음 처리 대상은 본인 알림만 가능
- Push 구독이 없는 사용자에게는 In-app만 생성

#### 제약조건
- MVP에서 Push는 **새 메시지 계열 우선**
- 북마크 알림은 운영 실험 플래그로 시작 가능
- 알림 클릭 시 원본 엔터티(대화방/포트폴리오 상세)로 이동 가능해야 함

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 알림 센터, 헤더 알림 아이콘, 메시지 관련 화면 참조
- 탭 구조/카운트 배지 위치는 와이어프레임 확인 필요

### 엣지케이스
- 같은 메시지 이벤트가 재시도 로직으로 중복 생성
- 원본 리소스가 삭제된 알림 클릭
- 브라우저 Push 권한 거부
- 다중 디바이스에서 읽음 동기화 지연

### 수용 기준 (Acceptance Criteria)
1. 새 메시지 수신 시 In-app 알림이 생성된다.
2. Push를 허용한 사용자에게는 메시지성 이벤트가 Push로도 전달된다.
3. 사용자는 본인 알림만 읽음 처리할 수 있다.
4. 알림 클릭 시 관련 화면으로 이동한다.
5. unread 알림 수는 헤더/알림센터와 일관되게 동기화된다.

## 2.9 북마크

### 기능명
북마크 (관심 포트폴리오 저장)

### 우선순위
MVP

### 사용자 스토리
- **구인자로서**, 나중에 다시 볼 후보를 저장하고 싶다, **왜냐하면** 한 번에 모든 연락을 결정하지 않기 때문이다.
- **비관여 방문자 또는 구직자로서**, 인상적인 포트폴리오를 모아두고 싶다, **왜냐하면** 재방문과 비교가 쉬워야 하기 때문이다.

### 상세 요구사항

#### 입력
- `portfolioId`
- 북마크 동작: `add` 또는 `remove`

#### 출력
- 현재 북마크 상태(`bookmarked: true/false`)
- 북마크 생성/삭제 시각
- 내 북마크 목록

#### 유효성 검증
- 자기 자신 포트폴리오 북마크 허용 여부는 정책 결정 필요
  - MVP 권장: 허용하되 UX적으로 노출 최소화
- 중복 북마크는 새 row를 만들지 않고 idempotent 처리
- 비공개/삭제 포트폴리오 북마크 추가 불가

#### 제약조건
- 목록은 최근 저장순 기본 정렬
- 북마크 해제는 즉시 반영
- 포트폴리오가 archived/deleted 되면 목록에서 숨기되, 사용자에게 "비공개/삭제" 상태 표시 가능

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 포트폴리오 상세, 카드 액션, 북마크 목록 화면 참조
- 북마크 아이콘 위치/토글 방식은 와이어프레임 확인 필요

### 엣지케이스
- 이미 저장된 항목 재저장
- 포트폴리오 삭제 후 북마크 목록 잔존
- 로그인 만료 상태에서 토글 시도
- 북마크 목록에서 대량 스크롤/재정렬

### 수용 기준 (Acceptance Criteria)
1. 로그인 사용자는 포트폴리오를 북마크 추가/해제할 수 있다.
2. 동일 포트폴리오를 여러 번 저장해도 중복 데이터가 생성되지 않는다.
3. 북마크 목록은 최근 저장순으로 조회된다.
4. 비공개 또는 삭제된 포트폴리오는 신규 북마크 추가가 불가능하다.
5. 북마크 상태는 카드/상세/북마크 목록에서 일관되게 보인다.

## 2.10 회원가입 / 로그인

### 기능명
회원가입 / 로그인 (이메일 + 소셜)

### 우선순위
MVP

### 사용자 스토리
- **신규 사용자로서**, 이메일이나 소셜 계정으로 빠르게 가입하고 싶다, **왜냐하면** 탐색과 연락을 빨리 시작하고 싶기 때문이다.
- **재방문 사용자로서**, 안정적으로 세션을 유지하고 싶다, **왜냐하면** 매번 재인증하는 마찰이 크기 때문이다.

### 상세 요구사항

#### 입력
- 이메일 회원가입:
  - `email`
  - `password`
  - `displayName`
- 소셜 로그인:
  - OAuth provider 1종 이상
- 선택 정보:
  - 역할(`assistant` / `recruiter`)

#### 출력
- 세션 생성
- 사용자 기본 프로필 행 생성
- 온보딩 필요 여부
- 실패 시 표준화된 에러 코드

#### 유효성 검증
- 이메일 형식 검사
- 비밀번호 8자 이상, 영문/숫자/특수문자 조합 권장
- 중복 이메일 가입 불가
- 이메일 미인증 사용자의 보호 기능(write actions) 제한 여부 정책화 필요
- OAuth 로그인 후 동일 이메일 매핑 정책 명시 필요

#### 제약조건
- MVP 기준 인증 방식:
  - 이메일/비밀번호
  - 소셜 OAuth 1종 이상
- 비밀번호 재설정 플로우 제공
- 로그아웃 시 서버/클라이언트 세션 동기 종료
- 인증 후 최초 1회 온보딩(role 선택 + 최소 프로필 작성) 수행

### 와이어프레임 참조
- `DrawMate/Wireframe/` 내 회원가입/로그인, 온보딩, 설정 화면 참조
- 소셜 버튼 배치/에러 상태 UI는 와이어프레임 확인 필요

### 엣지케이스
- 이메일 가입 후 인증 미완료 상태
- 동일 이메일로 소셜 로그인 시도
- 소셜 provider 콜백 실패
- 브라우저 쿠키 차단
- 비밀번호 재설정 링크 만료

### 수용 기준 (Acceptance Criteria)
1. 사용자는 이메일 또는 소셜 계정으로 가입/로그인할 수 있다.
2. 인증 성공 시 세션이 생성되고 최소 프로필이 준비된다.
3. 인증 실패 시 명확한 오류 메시지가 반환된다.
4. 로그아웃 시 보호 화면 접근이 차단된다.
5. 최초 로그인 사용자는 온보딩 단계로 유도된다.

## 3. 기능 간 의존성

| 선행 기능 | 후행 기능 | 이유 |
|---|---|---|
| 회원가입/로그인 | 프로필, 포트폴리오, 북마크, 메시지, 알림 | 대부분의 보호 기능은 인증이 선행되어야 함 |
| 사용자 프로필 | 포트폴리오 발행 | 작성자 정보가 비어 있으면 신뢰성이 낮아짐 |
| 이미지 업로드 | 포트폴리오 발행 | 발행 조건에 최소 1장 이미지 필요 |
| 태그 시스템 | 검색/필터링 | 구조화된 탐색을 위한 기본 데이터 |
| 포트폴리오 CRUD | 북마크, 메시지 | 연락과 저장의 대상이 포트폴리오 |
| 메시징 | 알림 시스템 | 새 메시지 이벤트의 주요 출처 |
